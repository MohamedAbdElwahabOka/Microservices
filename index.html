Client - Server

الـ Client هو أي حاجة بتطلب خدمة (موبايلك، المتصفح بتاعك). الـ Server هو الجهاز اللي بيرد على الطلب ده وبيديك الإجابة. زي ما تطلب أكل من ويتر (Client) وهو يجيبه من المطبخ (Server).

📨

API & HTTP Request

الـ API هو "قائمة الطلبات" المتاحة اللي أي برنامج تاني يقدر يطلبها من السيرفر بتاعك. الـ HTTP Request هو "الرسالة" الفعلية اللي بتتبعت لما تطلب حاجة من الـ API ده (زي: GET /api/orders).

🔌

Port (المنفذ)

لو السيرفر جهاز واحد بيشغّل أكتر من خدمة، الـ Port هو "رقم الباب" اللي كل خدمة بتسمع منه. مثلاً Orders Service بتسمع على الباب 3002، وInventory على الباب 3003 — كده أي طلب يعرف يوصل للخدمة الصح.

🧠

Stateless (بلا ذاكرة)

يعني الخدمة "منساها" أي حاجة بعد ما تخلص الطلب. كل Request بييجي لازم يجيب كل المعلومات اللي محتاجها معاه. الفايدة؟ تقدر تشغّل 5 نسخ من نفس الخدمة، وأي نسخة تقدر تخدم أي طلب من غير ما "تفتكر" حاجة عن الطلبات اللي قبلها.

🗂️

Database (قاعدة البيانات)

المكان اللي بيتخزن فيه الداتا بشكل دائم (مش زي الذاكرة اللي بتتمسح لو السيرفر قفل). في المقال ده هتشوف أنواع مختلفة: MongoDB, PostgreSQL, MySQL, Redis — كل نوع بيتفصل حسب طبيعة الداتا.

api-gateway/server.js
const express = require('express');
const httpProxy = require('http-proxy-middleware');
const app = express();

// الـ Gateway بيستقبل كل الطلبات من الـ Client
// وبيوجهها للخدمة الصح زي المدير اللي بيوزع الشغل

// لو الـ URL بيبدأ بـ /api/auth، يروح لـ Auth Service
app.use('/api/auth', httpProxy.createProxyMiddleware({
    target: 'http://auth-service:3001',
    changeOrigin: true,
}));

// لو الـ URL بيبدأ بـ /api/orders، يروح لـ Orders Service
app.use('/api/orders', httpProxy.createProxyMiddleware({
    target: 'http://orders-service:3002',
    changeOrigin: true,
}));

// لو الـ URL بيبدأ بـ /api/inventory، يروح لـ Inventory Service
app.use('/api/inventory', httpProxy.createProxyMiddleware({
    target: 'http://inventory-service:3003',
    changeOrigin: true,
}));

// لو الـ URL بيبدأ بـ /api/payment، يروح لـ Payment Service
app.use('/api/payment', httpProxy.createProxyMiddleware({
    target: 'http://payment-service:3004',
    changeOrigin: true,
}));

app.listen(8080, () => {
    console.log('🛡️ API Gateway running on port 8080');
});
orders-service/index.js
const express = require('express');
const axios = require('axios'); // عشان نكلم الخدمات التانية
const app = express();
app.use(express.json());

// Route بتاع إنشاء أوردر جديد
app.post('/api/orders', async (req, res) => {
    const { userId, productId, quantity } = req.body;

    try {
        // =====================================================
        // STEP 1: اسأل خدمة المخازن: "الكمية دي متاحة؟"
        // بنكلم Inventory Service مباشرةً عن طريق HTTP
        // =====================================================
        const stockCheck = await axios.get(
            `http://inventory-service:3003/api/stock/${productId}`
        );

        if (stockCheck.data.available < quantity) {
            return res.status(400).json({ 
                error: 'عفواً! الكمية المطلوبة غير متاحة في المخزن.' 
            });
        }

        // =====================================================
        // STEP 2: احفظ الأوردر في الداتابيز الخاصة بالـ Orders
        // =====================================================
        const newOrder = await Order.create({
            userId,
            productId,
            quantity,
            status: 'pending',
            createdAt: new Date()
        });

        // =====================================================
        // STEP 3: اخصم الكمية من المخزن
        // =====================================================
        await axios.patch(
            `http://inventory-service:3003/api/stock/${productId}`,
            { reduceBy: quantity }
        );

        // =====================================================
        // STEP 4: ابعت للـ Payment Service يعمل الدفع
        // =====================================================
        const payment = await axios.post(
            'http://payment-service:3004/api/charge',
            { orderId: newOrder.id, amount: stockCheck.data.price * quantity }
        );

        // =====================================================
        // STEP 5: ابعت إشعار للمستخدم (Fire and Forget)
        // مش هنستنى الرد، بس نبعتله الرسالة
        // =====================================================
        axios.post('http://notification-service:3005/api/notify', {
            userId,
            message: `تم شراء طلبك بنجاح! رقم الطلب: ${newOrder.id}`
        }).catch(err => {
            // لو الإشعارات عطلت، الأوردر هيتعمل عادي ومش هيتأثر!
            console.log('⚠️ Notification service is down, but order is saved.');
        });

        // =====================================================
        // SUCCESS: الأوردر اتعمل والفلوس اتخصمت!
        // =====================================================
        res.status(201).json({
            success: true,
            orderId: newOrder.id,
            paymentId: payment.data.paymentId,
            message: 'تم الطلب بنجاح! 🎉'
        });

    } catch (error) {
        res.status(500).json({ error: 'حصل خطأ، حاول مرة أخرى.' });
    }
});

app.listen(3002, () => console.log('🛒 Orders Service on port 3002'));
inventory-service/main.py (FastAPI)
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from database import get_db_connection  # اتصال بـ PostgreSQL

app = FastAPI()

# ملاحظة: الخدمة دي مكتوبة بـ Python
# بينما Orders Service مكتوبة بـ Node.js
# وده من أجمل حاجات الـ Microservices!

@app.get("/api/stock/{product_id}")
async def check_stock(product_id: int):
    """
    Orders Service بتكلمنا هنا عشان تعرف:
    1. الكمية المتاحة من المنتج
    2. سعر المنتج
    """
    db = get_db_connection()
    product = db.execute(
        "SELECT * FROM products WHERE id = ?", (product_id,)
    ).fetchone()

    if not product:
        raise HTTPException(status_code=404, detail="المنتج مش موجود!")

    return {
        "product_id": product_id,
        "name": product["name"],
        "available": product["stock_count"],
        "price": product["price"]
    }


class StockUpdate(BaseModel):
    reduce_by: int


@app.patch("/api/stock/{product_id}")
async def reduce_stock(product_id: int, update: StockUpdate):
    """
    Orders Service بتكلمنا هنا بعد ما الأوردر اتسجل عشان نخصم الكمية
    """
    db = get_db_connection()
    db.execute(
        "UPDATE products SET stock_count = stock_count - ? WHERE id = ?",
        (update.reduce_by, product_id)
    )
    db.commit()
    
    print(f"✅ Reduced stock for product {product_id} by {update.reduce_by}")
    return {"success": True, "message": "تم خصم الكمية من المخزن!"}


# شغّل السيرفر على بورت 3003

📦

العزل التام (Isolation)

كل خدمة في صندوقها، لو المخزن (Inventory Container) هنج، باقي الـ Containers شغالة. وتقدر تعمل docker restart inventory من غير ما حاجة تاني تتأثر.

🔄

الـ Scaling بسهولة

الكاشير (Orders) عليه ضغط؟ في ثانية واحدة تشغّل نسخ زيادة:
docker-compose up --scale orders=5
وبكده بيبقى عندك 5 Containers بيشتغلوا في نفس الوقت! (ده ممكن لأن الخدمة Stateless — أي نسخة تقدر تخدم أي طلب من غير ما "تفتكر" حاجة).

🚀

Deploy بسرعة البرق

عملت تعديل في الـ Payment Service بس؟ ابني الـ Image الخاص بيها لوحده وانشره:
docker build -t payment .
docker push payment
بدون ما تمس باقي الخدمات.

🌍

بيشتغل في أي مكان

اشتغل على الـ Mac، Staging على Ubuntu، Production على AWS. نفس الـ Image، نفس النتيجة. خلصت مشكلة "عندي بيشتغل"!

ازاي بنكتب Dockerfile؟

الـ Dockerfile هو "وصفة" بناء الـ Container. زي ما المطبخ عنده وصفة لكل أكلة.

FROM node:18-alpine # ← ابدأ من صورة Node.js جاهزة (زي قاعدة العجينة)
WORKDIR /app # ← شغّل كل الأوامر جوا فولدر /app
COPY package*.json ./ # ← انسخ ملفات الـ packages الأول
RUN npm install # ← ثبّت الـ packages جوا الـ Container
COPY . . # ← انسخ باقي كود المشروع
EXPOSE 3001 # ← افتح البورت 3001 للعالم
CMD ["node", "index.js"] # ← الأمر اللي هيشغّل السيرفر لما الـ Container يبدأ
FROM python:3.11-slim # ← صورة Python رفيعة وخفيفة
WORKDIR /app # ← فولدر الشغل
COPY requirements.txt . # ← ملف المكتبات المطلوبة
RUN pip install -r requirements.txt # ← ثبّت fastapi, sqlalchemy, إلخ
COPY . . # ← انسخ كود Python
EXPOSE 3003 # ← بورت الـ Inventory Service
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "3003"] # ← شغّل FastAPI
FROM golang:1.21-alpine AS builder # ← مرحلة البناء (Multi-stage build)
WORKDIR /app # ← فولدر البناء
COPY . . # ← انسخ كود Go
RUN go build -o payment . # ← compile الكود لـ binary صغير
FROM alpine:latest # ← مرحلة التشغيل (بدون Go، حجم أصغر بكتير!)
COPY --from=builder /app/payment . # ← خد الـ binary بس من مرحلة البناء
EXPOSE 3004 # ← بورت الـ Payment Service
CMD ["./payment"] # ← شغّل الـ binary

🤔 الأسئلة الشائعة

أسئلة كل developer بيسألها عن الـ Microservices

كل سؤال ممكن يخطر على بالك، هتلاقي إجابته هنا بالتفصيل الممل الجميل.

01 هبدأ بالـ Microservices من أول مشروع؟

❌ لأ، ده غلط شائع جداً! الـ Microservices مش مناسبة للمشاريع الصغيرة أو الـ Startups في أولها. السبب؟

  • الـ Microservices بتزود التعقيد بشكل هائل (Network calls، Distributed logging، Service discovery).
  • لو مش عندك فريق كبير ومتخصص، هتتعب من الـ overhead بتاع الـ DevOps.
  • شركات زي Amazon و Netflix نفسها بدأت Monolithic وبعدين انتقلت للـ Microservices لما كبرت.
الـ Rule of Thumb: ابدأ Monolithic، لما الفريق يكبر وجزء من التطبيق يحتاج Scaling مستقل، وقتها افصله كـ Microservice.
02 إزاي الـ Services بتعرف تتواصل مع بعض؟ في أكتر من طريقة؟

أيوه، في 3 طرق رئيسية للتواصل:

🌐

REST API (HTTP)

الأشهر والأبسط. Service بتكلم التانية عن طريق HTTP request زي ما الموبايل بيكلم السيرفر. المشكلة: لو الـ Service التانية بطيئة، المستدعي هيفضل مستنيها.

GET http://inventory-service/api/stock/7
📨

Message Queue (RabbitMQ / Kafka)

Service بتبعت رسالة في "صندوق بريد" (Queue)، والـ Service التانية بتاخدها لما تجهز. الإرسال Async ومش محتاج رد فوري. الأمثل للإشعارات والأحداث.

queue.publish("order.created", orderData)
⚡

gRPC

بروتوكول من Google. أسرع بكتير من REST لأنه بيستخدم Binary format بدل JSON. بيستخدم بشكل كبير في internal service-to-service communication.

stub.CheckStock(request)
03 لو Service وقعت، إزاي بنتعامل معاها؟ (Fault Tolerance)

ده من أهم التحديات. في 3 Patterns مشهورين لحل المشكلة دي:

Circuit Breaker

زي القاطع الكهربائي في بيتك. لو Service استدعيت 5 مرات وفشلت، الـ Circuit Breaker "يفتح" ويوقف الاستدعاء ويرجع response جاهز بدل ما يستنى ويعطّل التطبيق كله. بعد وقت معين، بيجرب يوصل تاني.

Retry with Backoff

لو الاستدعاء فشل، جرب تاني بعد ثانية، وتاني بعد 2 ثانية، وتاني بعد 4 ثانية. مش هتلحق على Service لقعانة ومش هتضيع وقت بدون داعي.

Fallback

لو Service المنتجات واقعة، ارجع بـ "cached data" قديم أو رسالة "المنتجات غير متاحة مؤقتاً" بدل ما التطبيق كله يقع. المستخدم هيحس بتأثير بسيط مش بـ Error ضخم.

04 إزاي بنتابع الـ Logs لما عندنا 10 Services مختلفة؟

ده من أصعب حاجات الـ Microservices. تخيل إن request من المستخدم عدى على 5 Services ومش عارف في أنهي واحدة حصل الـ bug. الحل بيبقى Distributed Tracing.

الفكرة: كل request بييجي بياخد Trace ID فريد (زي رقم الأوردر)، وكل service بتبعت اللوج بالـ ID ده لـ System مركزي.

📊 ELK Stack Elasticsearch + Logstash + Kibana — بيجمع اللوجات من كل الخدمات في مكان واحد
🔍 Jaeger / Zipkin بيرسم لك رحلة الـ Request خطوة بخطوة وبيقولك كل Service اخدت كام millisecond
📈 Grafana + Prometheus بيعمل Dashboards بتشوف فيها أداء كل Service لحظة بلحظة
05 إيه الفرق بين الـ API Gateway وعادي بيكلم الـ Service مباشرةً؟

الـ API Gateway هو "حارس البوابة" اللي الـ Client بيكلمه بدل ما يكلم كل Service على حدة. وده بيضيف:

  • 🔑 Authentication مركزي: بتتحقق من الـ Token مرة واحدة في الـ Gateway بدل ما كل Service تعمل نفس الكلام.
  • ⚖️ Load Balancing: لو عندك 3 نسخ من Orders Service، الـ Gateway هو اللي بيوزع الطلبات بينهم.
  • 🛡️ Rate Limiting: يمنع حد ما يعمل 1000 request في الثانية ويضرب السيرفر.
  • 📝 Logging مركزي: كل Request بيتسجل في مكان واحد.
  • 🔀 Routing: بيحول /api/orders لسيرفر الأوردرات و/api/auth لسيرفر الـ Auth.
أشهر الـ API Gateways: NGINX، Kong، AWS API Gateway، Traefik
06 إزاي بنعمل Transaction واحدة بتاخد من 3 داتابيزات مختلفة؟

ده من أصعب الأسئلة في الـ Microservices! في الـ Monolithic، عندنا Database Transaction بتضمن إما كل حاجة تتعمل أو ما اتعملش حاجة (All or Nothing) وبتحصل فوراً. في الـ Microservices ده صعب لأن كل خدمة ليها داتابيز مستقلة، فمفيش transaction واحدة تشمل الكل.

الحل هنا مبني على فكرة اسمها Eventual Consistency (الاتساق النهائي): يعني بدل ما الداتا تتزامن فوراً في نفس اللحظة، بتوصل لحالة متسقة "بعد شوية" عن طريق سلسلة خطوات، وكل خطوة فشلت بيتعمل لها "تراجع" (Rollback) بخطوة عكسية. الـ Pattern الشائع لتنظيم السلسلة دي اسمه Saga Pattern:

1
Orders Service: حفظت الأوردر بحالة "Pending" ✅ نجح → ابعت event: order.created
↓
2
Inventory Service: خصمت الكمية من المخزن ✅ نجح → ابعت event: stock.reserved
↓
3
Payment Service: فشل الدفع! البطاقة مرفوضة ❌ فشل → ابعت event: payment.failed
↓ (Compensating Transactions)
↩
Inventory: رجّعت الكمية للمخزن تاني
Orders: غيّرت حالة الأوردر لـ "Failed" ↩ كل حاجة رجعت لحالتها الأصلية
07 Docker Compose بيكفي ولا لازم Kubernetes؟

الفرق جوهري:

🐳 Docker Compose

  • مناسب للـ Development وTesting
  • بيشغّل Containers على جهاز واحد بس
  • مش بيعمل Auto-healing (لو Container وقع مش هيرجعه)
  • الـ Scaling يدوي وبدائي
  • سهل وبسيط للبداية ✅

☸️ Kubernetes (K8s)

  • مناسب للـ Production الجاد
  • بيوزع الـ Containers على آلاف الأجهزة
  • Auto-healing: لو Container وقع، K8s بيعيد تشغيله تلقائياً
  • Auto-scaling: الضغط زاد؟ بيفتح Containers جديدة لوحده
  • معقد ومحتاج تعلم منفصل
الخلاصة: ابدأ بـ Docker Compose. لما تروح للـ Production الجاد وعندك ملايين users، وقتها انتقل لـ Kubernetes أو استخدم managed service زي AWS ECS / Google Cloud Run.
08 إزاي بنعمل Testing للـ Microservices؟

التست في الـ Microservices على 4 مستويات:

Unit Tests بتختبر Function واحدة

تختبر كل function في العزل الكامل. سريع جداً وبيشتغل بدون داتابيز أو Network.

expect(calculateTotal(items)).toBe(500)
Integration Tests بتختبر Service كاملة مع داتابيزها

بتشغّل الـ Service مع الداتابيز الحقيقية (أو test database) وبتختبر الـ API Endpoints.

POST /api/orders → expect(201, {orderId})
Contract Tests بتختبر التواصل بين الخدمات

بتتأكد إن لو Orders Service بتبعت Request بشكل معين، الـ Inventory بتفهمه صح. أداة مشهورة: Pact.

E2E Tests بتختبر رحلة المستخدم كاملة

بتشغّل كل الخدمات مع بعض وبتعمل طلب شراء كامل من الأول للآخر وبتتأكد إن كل حاجة شغالة. بطيء لكن بيكشف مشاكل التكامل.

09 إزاي بنعمل الـ Security في الـ Microservices؟

الـ Security في الـ Microservices بيتعمل على مستويين:

🛡️

External Security (Client → Gateway)

  • JWT Token: Client بيبعت Token مع كل Request، الـ Gateway بيتحقق منه قبل ما يوصل لأي خدمة.
  • HTTPS: كل الـ Traffic مشفّر.
  • Rate Limiting: الـ Gateway بيمنع Brute Force attacks.
🔐

Internal Security (Service → Service)

  • mTLS: كل خدمة بيكون عندها Certificate وبتتحقق من هوية الخدمة التانية.
  • Service Mesh (Istio): بيعمل Encryption تلقائي لكل الـ internal communication.
  • Network Policies: تحدد إن Orders Service بتكلم Inventory بس، ومش بتكلم الـ DB مباشرةً.
10 شركات زي Netflix و Amazon بتعمل Microservices إزاي فعلاً؟

أرقام حقيقية هتبهرك:

🎬

Netflix

  • أكتر من 700 Microservice
  • بيعملوا أكتر من 1 مليون deployment في السنة
  • بيستخدموا Chaos Engineering — بيطفوا Services عمداً في Production عشان يتأكدوا إن النظام بيتحمل!
  • بيستخدموا أداتهم الخاصة Hystrix للـ Circuit Breaker
🛒

Amazon

  • بدأوا Monolithic في التسعينيات
  • انتقلوا للـ Microservices في بداية الألفينيات
  • كل Service يملكها فريق صغير (Two-Pizza Team — فريق يتغذى على بيتزتين!)
  • الـ Microservices architecture بتاعتهم أدى لاختراع AWS نفسه — لأنهم احتاجوا infrastructure مرن!
🚕

Uber

  • بدأوا بـ Monolithic بلغة Python
  • انتقلوا لأكتر من 2000 Microservice
  • بعدين اكتشفوا إن ده كتير جداً وبدأوا يدمجوا بعضهم! (Macro-services)
  • الدرس: الـ Microservices مش "أكتر = أحسن"
الخلاصة الذهبية: حتى الشركات الكبيرة دي غلطت وتعلمت. مفيش "right answer" واحد — الصح هو اللي يناسب حجم مشروعك وفريقك.
11 إزاي النظام بيعرف إن خدمة "حية" وشغالة كويس؟ (Health Checks)

ده أبسط وأهم مفهوم قبل ما تتكلم عن Monitoring متقدم زي ELK أو Grafana. كل خدمة كويسة بيكون عندها Endpoint بسيط اسمه غالباً /health، بيرجع بس رسالة صغيرة تقول "أنا شغالة":

💚

Health Check Endpoint

مثال بسيط: أي حد (أو أي نظام) يعمل GET على الرابط ده، لو الخدمة شغالة هترجع 200 OK. لو مش شغالة، الطلب هيفشل أو يرجع خطأ.

GET /health → {"status": "ok"}
🔁

مين بيسأل السؤال ده؟

الـ Load Balancer أو Kubernetes بيسأل الخدمة كل كام ثانية "إنتي لسه شغالة؟". لو مردتش لعدد مرات معين، النظام بيعتبرها "واقعة" ويوقف إرسال طلبات ليها، وأحياناً بيعيد تشغيلها تلقائياً.

من غير Health Checks، النظام مش هيعرف إن خدمة معينة "متجمدة" (Hanging) لسه شغالة تقنياً بس مش بترد فعلياً — وده أخطر من إنها تقع تماماً!
